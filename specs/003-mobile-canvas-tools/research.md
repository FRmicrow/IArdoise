# Research: Mobile Fullscreen Canvas & Drawing Tools

## 1. Mobile fullscreen canvas

**Decision**: Replace `100vh` with `100dvh` (falling back to `100vh` for browsers that don't understand `dvh`) on `#app` and `.page--full` in `frontend/src/style.css`. Add `viewport-fit=cover` to the viewport `<meta>` tag in `frontend/index.html` and safe-area-inset padding on `.page--full` so the header/canvas aren't obscured by notches or rounded corners.

**Rationale**: The current code (`frontend/src/style.css:54,81`) sizes the player's full-screen layout with static `100vh`. Static `vh` is computed against the *largest* possible viewport and does not shrink when a mobile browser's address bar is visible, nor grow back when it collapses — this is the textbook cause of clipped or gapped mobile canvases. `dvh` (dynamic viewport height) recomputes as the browser chrome shows/hides and is supported in the project's actual target browsers (iOS Safari 16+, Chrome/Android 108+, both far below what `vite-plugin-pwa`'s installable-PWA target requires). CSS silently ignores an unknown unit, so declaring `height: 100vh;` first and `height: 100dvh;` second is a safe fallback with zero JS.

Confirmed empirically relevant: `frontend/src/pages/game-player.ts:13-20` already hides the `#waiting` banner (`display: none`) once the round is active, and `.page--full`'s `grid-template-rows: auto auto 1fr` (`style.css:80-87`) collapses an `auto` row with no visible content to ~0 height. So once a round is active, the layout already resolves to "phrase header + canvas" visually — the outstanding gap is the viewport-unit bug, not the row structure. No markup restructuring is required to satisfy "only the question as header, the rest is canvas"; the CSS unit fix is sufficient. This will be verified with a Playwright assertion (canvas bounding box height vs. `window.visualViewport.height`) on the `Mobile Safari` project already configured in `frontend/playwright.config.ts:23-26`.

**Alternatives considered**:
- JS-driven `--vh` custom property (`window.visualViewport` / `resize` listener setting a CSS variable), the common pre-`dvh` workaround. Rejected: adds a resize-event listener and a layout-thrashing risk for a problem the CSS `dvh` unit already solves natively in every target browser — extra JS for a solved problem violates Constitution Principle V (Minimal-Footprint, YAGNI).
- Restructuring `.page--full` into a 2-row grid (`auto 1fr`) instead of `auto auto 1fr`. Rejected: unnecessary churn — the existing 3-row grid already collapses correctly when the waiting banner is hidden (see above); changing it would only be relevant if the waiting banner needed to coexist with the canvas, which it doesn't per the existing state machine.

## 2. Drawing engine: Fabric.js

**Decision**: Add `fabric` (npm package, current major v7, `7.4.0` at time of writing) as a new frontend dependency and rebuild `frontend/src/canvas/DrawingCanvas.ts` on top of `fabric.Canvas` with `PencilBrush` for freehand drawing, plus the companion `@erase2d/fabric` package (`1.2.1`, peer-dependency `fabric >= 6.0.0`) for the eraser tool.

**Rationale**: Explicit stakeholder decision (recorded in `spec.md` Assumptions) to use Fabric.js rather than hand-rolling color/width/eraser on the existing raw Canvas 2D code in `DrawingCanvas.ts`. Verified against current Fabric.js docs (Context7, `/websites/fabricjs` and `/fabricjs/fabricjs.com`) and the published npm packages, not assumed:
- Free drawing: `new Canvas(el, { isDrawingMode: true })` + `canvas.freeDrawingBrush = new PencilBrush(canvas)`; brush `color` and `width` are plain settable properties, read directly from the toolbar's color/size controls before each stroke.
- Erasing: as of Fabric.js v6, `EraserBrush` was extracted out of the core `fabric` package into a separate maintained package, `@erase2d/fabric` (confirmed: `fabric@7.4.0`'s shipped `dist/index.d.ts` has no `Eraser` export at all; `@erase2d/fabric@1.2.1` explicitly declares `peerDependencies: { fabric: ">=6.0.0" }` and exports `EraserBrush`). The eraser tool must import `EraserBrush` from `@erase2d/fabric`, not from `fabric` — using the older `fabric.EraserBrush` API shown in pre-v6 demos would silently fail to erase anything on this project's v7 install. Toggling to eraser mode is `canvas.freeDrawingBrush = new EraserBrush(canvas)` with `canvas.isDrawingMode` left `true`; toggling back to drawing swaps `freeDrawingBrush` back to the `PencilBrush` instance (with its last-selected color/width preserved).
- Fabric v6+ ships its own TypeScript types in the `fabric` package itself; the separate `@types/fabric` package on npm (latest `5.3.11`) targets the old v5 API shape and must **not** be installed alongside `fabric@7.x` — installing both would introduce a conflicting/duplicate type definition for the same import.

**Rendering mechanism (Constitution compliance check)**: The Frontend & UX Standards principle requires 60fps rendering specifically "via `requestAnimationFrame`." Verified via Context7 (`/fabricjs/fabricjs.com`, source excerpt of `Canvas.requestRenderAll`): Fabric.js schedules all rendering — including live free-drawing brush strokes — through `requestRenderAll()`, which calls `fabric.util.requestAnimFrame` (Fabric's `requestAnimationFrame` wrapper) and ignores redundant requests already pending for the current frame. This is the same scheduling primitive the constitution mandates, not a different one — the migration doesn't trade away the rAF-based rendering discipline, it inherits an equivalent one from the library. Actual frame-budget cost (does Fabric's fuller object model keep pace with the existing hand-rolled code's headroom) is a separate, empirical question, tracked as a manual check in tasks.md (SC-006).

**Bundle/offline impact**: `frontend/vite.config.ts`'s `vite-plugin-pwa` config caches JS/CSS via a runtime `NetworkFirst` rule (`urlPattern: /\.(?:js|css|...)$/`, `vite.config.ts:45-52`), not a static precache manifest — so the additional Fabric.js/erase2d JS chunk is cached on first load like any other build asset, with no service-worker config change required. Bundle size grows measurably (Fabric.js is a full-featured object-model canvas library); this is an accepted, explicit tradeoff per the stakeholder's direction and is called out in `plan.md`'s Complexity Tracking as an exception to Constitution Principle V.

**Alternatives considered**:
- Extending the existing hand-rolled `DrawingCanvas.ts` in place (add a `colour`/`width` parameter to `pointerdown`, and implement an eraser via `globalCompositeOperation = 'destination-out'` stroke segments). Technically viable for color/width (already trivial — `strokeColour`/`width: 4` are already per-stroke fields on `Stroke`, just hardcoded); erasing is the hard part, since a robust "erase what's under the pointer" needs either destructive pixel compositing (loses the redraw-on-resize capability the current code relies on, `DrawingCanvas.ts:145-153`, since composited erasure isn't stored as replayable vector data) or hand-rolled hit-testing against stored strokes. Rejected per the stakeholder's explicit direction to use a library, and because Fabric.js's object model (vector `Path` objects, each independently erasable/re-renderable) solves exactly this replay/resize problem the current code already depends on.
- Other canvas libraries (Konva.js, Paper.js). Not evaluated in depth — the stakeholder named Fabric.js specifically; no reason to deviate.

## 3. Toolbar UI placement

**Decision**: A small overlay toolbar (color swatch/input, a width slider, a draw/erase toggle) positioned with `position: absolute` inside `.canvas-container`, using the project's existing CSS custom-property tokens (`--color-*`, `--space-*`, `--radius-*` already used throughout `style.css`) — no new hardcoded colors/px.

**Rationale**: FR-009 requires tool controls reachable without shrinking the drawable canvas area. `.canvas-container` (`style.css:259-265`) is already `position`-able (it's a block-level flex/grid child); the existing `.overlay` pattern (`style.css:274-285`, used for the host-disconnected message) is the established precedent in this codebase for absolutely-positioned UI over the canvas. Reuses that pattern instead of inventing a new one. Constitution's Frontend & UX Standards mandates design-token-only styling, so the toolbar consumes existing tokens rather than introducing new hex/px literals.

**Alternatives considered**: A persistent non-overlay toolbar row added to the `.page--full` grid (a 4th row). Rejected: shrinks the canvas's `1fr` row, directly conflicting with FR-009 and the "all remaining space is canvas" requirement from the spec.
