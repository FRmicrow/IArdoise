## Manual regression check for this fix

The bug only reproduces when a player's socket is silently dead (no `close`
event fires on either end) while the tab stays foregrounded. This can't be
triggered by simply putting a laptop to sleep or backgrounding a tab (that
already worked before this fix, via `visibilitychange`). To reproduce and
verify the fix:

1. Start the app, create a session as host, and join as a player from a
   second browser/device so it lands on the waiting screen.
2. With the player's tab **kept in the foreground**, simulate a silently dead
   socket — e.g. via devtools: `Network` → throttle to `Offline` **without**
   closing the tab or triggering a visibility change, or by killing the
   underlying TCP connection at the OS/network level (e.g. disabling Wi-Fi
   without locking the screen). The key condition: no `close`/`error` event
   should fire on the player's `WebSocket` object, and the tab must stay
   visible the whole time.
3. On the host, click "Commencer la partie".
4. Restore the player's connectivity (re-enable Wi-Fi / remove the network
   throttle) without ever backgrounding the tab.
5. **Before the fix**: the player stays on the waiting screen indefinitely.
   **After the fix**: within ~15s of connectivity being restored (bounded by
   the heartbeat interval + reply timeout), the player's client detects the
   dead `PING`/`PONG` round trip, reconnects, and lands on the game screen
   with the current phrase — matching spec SC-003.
6. Repeat with a healthy connection throughout (no simulated drop) and
   confirm the transition still happens within ~3s, with no extra
   `PLAYER_DISCONNECTED`/`PLAYER_RECONNECTED` flicker on the host's roster
   (SC-002, FR-006 — no regression for the already-working path).
